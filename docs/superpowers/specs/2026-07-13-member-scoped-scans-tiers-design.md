# Member-scoped scans, tiered quotas & automated daily engine — Design Spec

- **Date:** 2026-07-13
- **Product:** Project Arklight (`arc-signal-core`)
- **Status:** Draft for owner review
- **Author:** Claude (architect), with Murray Hewitt-Coleman (owner)

---

## 1. Background & problem

### 1.1 The trigger
The operator reported: *"Scan buttons are not running new scans when selected."* A full diagnostic
found:

- **Every button in the app is correctly wired** (75 raw `<button>` tags, 100 `onClick`
  handlers, 7 `<Button>` components; zero dead/no-op/placeholder handlers). The only unwired
  element is `IconBtn` in `AppShell.tsx:242`, which is defined but never rendered (dead code).
- The scan chain is intact: **Run scan** button → `runScan()` server fn → `requireOwner`
  middleware → `runScanImpl()`.
- The failure is **not a code bug** in the wiring. It is the **owner-token gate**:
  - Client (`owner-auth-attach.ts`) sends `localStorage.owner_token` as the `x-owner-token` header.
  - Server (`owner-auth.server.ts` → `requireOwner`) compares it to `process.env.OWNER_TOKEN`;
    on mismatch or missing token it throws `Forbidden`.
  - The **only** place the token is ever *set* is a `window.prompt()` on the Narrative
    Divergence page (`narrative-divergence.tsx:34-37`). The scan buttons only *read* it. So a
    cleared/missing/stale token (or a server whose `OWNER_TOKEN` was reset by a redeploy) makes
    every owner-gated action silently fail with a "Scan failed: Forbidden" toast.

### 1.2 What it grew into
Rather than patch the token, the owner chose to redesign how scans work as a **member-facing,
tiered product capability**, with the heavy global engine automated on a schedule. This spec
captures that design.

### 1.3 Verification item carried from diagnosis
The existing cron job (`archlight-scan-6h`, migration `20260706042808…`) posts to
`https://project--frevemxrjbyngksweopd-dev.lovable.app/api/public/hooks/scan` — a **`-dev`
domain**, not the production site. **Implementation must confirm the cron hits the production
deployment**; if the dev URL is stale, automated scans have not been updating production, which
would independently explain "scans not running." (See Risks §9.)

## 2. Goals & non-goals

### Goals
1. Retire the fragile shared owner-token; gate operator/admin actions by **role** instead
   (fixes the reported bug).
2. Make **adding a book/watchlist item** immediately surface relevant signals (no dead feeling).
3. Give members a **"Scan my items"** action that fetches fresh data for *their* tracked
   entities and folds it into the shared graph, rate-limited by tier.
4. Keep the **full global engine** on its existing automated cron (**every 6 hours, 4×/day** —
   `'0 */6 * * *'`), which also refreshes every member's items and stamps them with a "last
   refreshed" time; confirm the cron targets the production deployment.
5. Move **Digest** and **Narrative Divergence** generation into the daily cron (automated
   outputs members read, not manual owner actions).
6. Keep **per-feature quotas separate** (no shared pool) with clear, distinct user-facing labels.

### Non-goals
- Live billing / Stripe checkout wiring (billing is not live; tiers resolve from existing data,
  defaulting everyone to free).
- Any change to production marketing routes or the Pygar project.
- Building a general multi-tenant permission system beyond the existing `user_roles` / tier model.

---

## 3. The model (four categories)

### ① Automated — existing cron every 6 hours (4×/day), system-run, free to all, no quotas
1. **Global scan** — collects world data, updates *every* member's book + watchlist items, and
   stamps each refreshed item with `last_refreshed_at`. The manual top-nav **"Run scan"** button
   is **removed** (redundant now the cron runs it) and replaced by a global freshness timestamp —
   see §4.8. Cadence stays every 6 hours (`'0 */6 * * *'`) — owner decision 2026-07-13.
2. **Digest** — generated by the cron; members read it.
3. **Narrative Divergence** — analysis runs in the cron; members read the results.

### ② Member manual actions — each with its own separate, clearly-named allowance
| Action | Free | Pro | Reset |
|---|---|---|---|
| **Scan my items** (new scoped fetch) | 3 / day | 10 / day | 00:00 GMT daily |
| **Research / Interrogate** | 1 / day | 60 / month | free: 00:00 GMT daily · pro: calendar month |
| **Ask the Graph** | 5 / day | 25 / day | 00:00 GMT daily |

### ③ Instant & free for members
- Adding/editing a book or watchlist item → immediate re-match against existing events.
- Reading digest, divergence, track record, exposures, etc.

### ④ Operator/admin only — gated by role (`has_role('admin', …)`), not the owner token
- Sources, scan settings, delivery channels, review queue, model routing, backtest harness.
- The manual "run global scan now" is **removed from the top nav** (replaced by the freshness
  timestamp, §4.8); an optional admin-only trigger may remain on the operator `/scans` page as a
  testing fallback (owner to confirm — §4.8).

## 4. Detailed design

### 4.0 Part D — Retire owner-token, gate by role *(do first; fixes the reported bug)*
- Introduce a `requireAdmin` (or `requireRole('admin')`) server middleware that reads the
  authenticated Supabase user (already available via `attachSupabaseAuth` / `requireSupabaseAuth`)
  and checks `has_role('admin', userId)` (RPC already exists).
- Replace `.middleware([requireOwner])` on **operator-only** server fns with `requireAdmin`.
- Delete the client `attachOwnerToken` middleware, the `owner-auth-attach.ts` /
  `owner-auth.server.ts` pair, the `window.prompt()` token flow, and the `OWNER_TOKEN` env
  dependency once all call sites are migrated.
- Seed at least one `user_roles` row (`role = 'admin'`) for the operator account.
- **Acceptance:** logged-in admin can run a manual scan with no token prompt; a non-admin cannot.

### 4.1 Part A — Instant re-match on add/edit item *(small; reuses `scoreExposures`)*
- On `addExposureItem` / `updateExposureProfile` / `removeExposureItem` and watchlist
  create/update, invoke a **scoped** `scoreExposures` pass for that user's profile against
  **existing** recent events (not only events from a scan run).
- `scoreExposures` currently keys off a `scanRunId` (scores events created in that run). Extend
  it to accept a mode that scores a **single profile** against a recent-events window, so a newly
  added item is matched immediately.
- **Acceptance:** adding "Acme" to a book produces its exposure hits without waiting for a scan.

### 4.2 Part B — "Scan my items" scoped fetch → folds into global graph *(main new build)*
Composition of two existing halves:

**Collection front-end (reuse Interrogate machinery):** for each of the member's tracked
entities (book + watchlist items), run `classifySubject` → `buildQueries` → `fetchGoogleNews`
(free, no API key), dedup links. This already exists in `interrogate` (`pipeline.functions.ts`).

**Persistence back-end (reuse global-scan pipeline):** feed the collected documents through the
**same** extract → cluster → dedup → persist path `runScanImpl` uses, so new claims/events land
in the shared tables and pass normal clustering (no duplicate events). Then run Part A's
re-match for the member.

- New server fn `scanMyItems` gated by `requireSupabaseAuth`, quota-checked (see §4.5).
- Writes go through the service/admin client (same as `runScanImpl`), not the member's RLS
  context, since results are global.
- Tag the resulting `scan_runs` row with `triggered_by = userId` and `trigger_kind = 'member_scoped'`.
- **Acceptance:** a member scan fetches only their entities' data, new events appear in the global
  graph, their exposure hits update, and one unit is deducted from their daily allowance.

### 4.3 Part C — Global cron (keep 4×/day) + automated digest/divergence
- **No reschedule.** Keep the existing pg_cron job `archlight-scan-6h` at `'0 */6 * * *'` (every 6
  hours, 4×/day) — owner decision 2026-07-13. The only cron fix required is to **confirm/repoint
  the hook URL to the production deployment** (§1.3), since it currently targets a `-dev` domain.
- Extend the scan hook (or a sibling hook) so each run **also** calls `generateDigest` and
  the divergence analysis (`autoAnalyseTopConvergence`) after the scan completes — all in the
  system/service context (no owner token).
- The global scan's exposure-scoring pass stamps `last_refreshed_at` on every item it re-scores.
- **Acceptance:** at 06:00 GMT the scan, digest, and divergence run automatically against
  production; member items show an updated "last refreshed" time.

### 4.4 Timestamps — `last_refreshed_at`
- `exposure_items` and `watchlists` already have `created_at` / `updated_at`, but `updated_at`
  reflects **user edits**, not scan refreshes — reusing it would mislead.
- Add a dedicated **`last_refreshed_at timestamptz`** column to `exposure_items` and `watchlists`
  (nullable), set whenever that item is re-scored by the daily cron or a member scan.
- Surface in the UI as *"Updated: 13 Jul, 06:00 GMT."*

### 4.5 Tiers & quota enforcement
- **Tier resolution:** reuse `isProUser(userId)` (`billing.functions.ts:56`), backed by the
  `subscriptions` table. No subscription / not pro → **free**.
- **Counting:** mirror the proven Interrogate pattern (count rows in a time window per user).
  - *Scan my items:* count `scan_runs` where `triggered_by = userId AND trigger_kind='member_scoped' AND started_at >= <00:00 GMT today>`; free cap 3, pro cap 10.
  - *Research:* free = count today's `investigation_queries` for the user (cap 1); pro = count this
    calendar month (cap 60).
  - *Ask the Graph:* record each call in the new `member_action_usage` table
    (`user_id, action='ask_graph', created_at`); count today's rows (free 5, pro 25).
- **Reset windows:** daily = 00:00 GMT; monthly = calendar month (GMT).
- **Over-limit UX:** friendly message + reset time + upgrade prompt; do not throw a raw error.
- **A single generic quota helper** (`assertWithinQuota({ userId, action })`) should encapsulate
  tier lookup + windowed count + limit table, so every gated action uses one code path and limits
  live in one config object.

### 4.6 Cost guardrails
- Cap **entities per member scan** (e.g. process at most N of the member's items per run; if they
  have more, process highest-priority first and note the rest were skipped — never silently
  truncate).
- Cap **fetches per run** (shared bounded budget like `runScanImpl`'s `bodyBudget`).
- Log skipped/uncovered items so the member sees what was and wasn't refreshed.

### 4.7 UX & copy *(drafted; requires owner sign-off per SR4 — never auto-applied)*
- **Pre-scan confirmation dialog:**
  > "Add every company, holding, sector and watchlist item you want covered before you scan — this
  > run only fetches what's on your list right now. You have **N scans left today**."
- **Persistent indicator:** "Scans left today: N" (and "Research runs left today/this month: N",
  "Graph queries left today: N" on their respective surfaces).
- **At limit:** button disabled with "Resets at midnight GMT · Upgrade to Pro for more."
- **Item freshness:** "Updated: {last_refreshed_at}" on each book/watchlist item.
- Copy must follow house rules: no em dashes in reader-facing copy, no AI-writing tells,
  benefit-first, GBP, "structured perspectives" not "coaching," no financial advice.

### 4.8 Top-nav global freshness timestamp *(replaces the manual "Run scan" button)*
Since the global scan is now automated by the daily cron, the top-nav **"Run scan"** button in
`AppShell.tsx` (`TopNav`) is **removed**. In its place, show a **global update timestamp** — the
completion time of the most recent successful global scan — visible to all users as a data-freshness
signal, e.g. *"Engine updated: 13 Jul, 06:00 GMT"* (optionally with a relative "· 2h ago").
- **Source:** latest `scan_runs` where status ∈ {completed, ok}; use `finished_at` (fallback
  `started_at`). Small dedicated query, or extend `getDashboard`.
- The `AppShell` `onRunScan` / `scanning` props (currently passed by `app.tsx` and `scans.tsx`) are
  **removed** from the top nav.
- The **member "Scan my items"** button (§4.2) is a *separate* control on member surfaces
  (dashboard / book / watchlist), not the top nav.
- **Existing `/scans` page "Run scan now" button:** default is to keep it as an admin-only
  (`requireAdmin`) manual fallback for testing; alternatively remove it. Owner to confirm.
- Removing this button also removes the last dependency on the client `attachOwnerToken` flow for
  scanning, complementing Part D.

## 5. Data / schema changes (Supabase migrations, via the normal Lovable flow)
1. `exposure_items.last_refreshed_at timestamptz null`
2. `watchlists.last_refreshed_at timestamptz null`
3. `scan_runs.triggered_by uuid null` + `scan_runs.trigger_kind text default 'system'`
   (values: `system`, `member_scoped`, `admin_manual`).
4. `member_action_usage` table (or equivalent) for Ask-the-Graph (and any action not otherwise
   countable): `id, user_id, action text, created_at timestamptz default now()` + index on
   `(user_id, action, created_at)`.
5. Cron: **no reschedule** — keep `archlight-scan-6h` at every 6 hours. Verify/repoint its hook URL
   to the **production** deployment (currently a `-dev` domain).
6. Seed `user_roles` admin row for the operator.

*(All migrations follow the existing `supabase/migrations/*` convention and apply through Lovable.)*

## 7. Component boundaries (for isolation & testability)
- **`quota` module** — one place: tier lookup + windowed counting + limit config + friendly
  over-limit messages. Consumed by every gated action.
- **`memberScan` module** — orchestrates: gather member entities → collect → persist (delegates to
  shared pipeline) → re-match → stamp timestamps → record usage.
- **`requireAdmin` middleware** — role gate; replaces `requireOwner`.
- **`rematch` function** — Part A; one profile vs recent events; reused by add-item and member scan.
- Each unit: clear single purpose, well-defined inputs/outputs, independently testable.

---

## 8. Security considerations
- Retiring the shared `OWNER_TOKEN` removes a fragile secret and a client-readable gate.
- All member actions run under `requireSupabaseAuth`; writes to global tables use the service
  client deliberately and only after quota + validation checks.
- The public cron hook must remain secret-gated (currently the project key); confirm the secret is
  a server env var that survives redeploys.
- Guardrails (`guardFinancialAdvice`) already applied in synthesis must remain on any member-facing
  generated text.

---

## 9. Risks & open items
1. **Cron `-dev` URL (§1.3)** — verify/repoint to production; may be a current live cause of stale
   data. High priority to confirm during Part C.
2. **Billing not live** — tiers resolve but nobody is "pro" until subscriptions exist; free caps
   effectively apply to all. Acceptable for launch; revisit when billing ships.
3. **Cost at scale** — many members × scoped scans × entities. Guardrails in §4.6 mitigate; monitor
   `llm_task_logs` cost.
4. **Duplicate events** — member-scoped writes must pass the same clustering/dedup as global scans;
   verify no duplicate `event_candidates`.
5. **Concurrency** — member scans and the daily cron both write globally; confirm the existing
   `already_running` / stale-reap logic behaves when a member scan overlaps the cron.
6. **RLS** — new columns/tables need appropriate policies; run `get_advisors(security)` after DDL.

## 10. Phasing / sequencing
1. **Phase 0 (quick win):** Part D — role gate for remaining operator actions; **remove the broken
   top-nav "Run scan" button and replace it with the global freshness timestamp (§4.8)**. This
   directly resolves the reported "scan button doesn't work" by retiring the button and cutting the
   owner-token dependency.
2. **Phase 1:** Part A — instant re-match on add/edit item.
3. **Phase 2:** quota module + schema (usage tracking, `last_refreshed_at`).
4. **Phase 3:** Part B — "Scan my items" scoped fetch.
5. **Phase 4:** Part C — keep 4×/day cron (no reschedule) + add digest/divergence to the automated run + verify/fix the production hook URL.
6. **Phase 5:** UX, copy, freshness timestamps, per-feature counters (with owner copy sign-off).

---

## 11. Out of scope
- Stripe checkout / live billing.
- Opening additional operator tools to members beyond those specified.
- Any Pygar-project or production-marketing changes.
