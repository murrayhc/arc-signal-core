# Multi-model LLM routing

Phase 3d. How Archlight routes optional LLM-assisted enrichment tasks across
providers/models, why it stays **dormant by default**, and how every LLM
output is validated before it can touch persisted data.

Source: `src/server/llm/router.ts`, `src/server/llm/provider.ts`,
`src/server/llm/run.ts`, `src/server/llm/validate.ts`,
`src/server/playbook/service.ts`, `src/server/services/playbook.ts`.

## 1. Why this exists

Every core Archlight surface (events, opportunities, positioning, playbooks)
is generated by a **deterministic template pipeline** — no LLM call is ever
required to produce a usable result. LLM orchestration is an optional,
strictly-validated *upgrade path* layered on top: when a provider is active
and its output clears every safety gate, a small number of text fields get
richer, evidence-grounded phrasing. When it isn't (the default), the
deterministic output is what ships. Nothing in the product depends on a live
model.

## 2. Task-type → model routing

`LLMTaskType` (`src/shared/enums.ts`) enumerates every task the system knows
how to route:

| Task type | Used by |
|---|---|
| `CLAIM_EXTRACTION_ASSIST` | claim extraction (assist-only) |
| `ENTITY_RESOLUTION_ASSIST` | entity resolution (assist-only) |
| `SIGNAL_CLASSIFICATION_ASSIST` | signal classification (assist-only) |
| `CONTRADICTION_ANALYSIS` | evidence-arc contradiction detection |
| `EVIDENCE_ARC_SUMMARY` | evidence-arc narrative summary |
| `STRATEGIC_POSITIONING_GENERATION` | positioning-example enrichment |
| `OPPORTUNITY_PLAYBOOK_GENERATION` | **playbook enrichment (this doc's focus)** |
| `EXECUTIVE_BRIEF_GENERATION` | executive-brief rendering |
| `GRAPH_NODE_SUMMARY` / `GRAPH_EDGE_EXPLANATION` | living-intelligence-graph enrichment |
| `MARKET_CONTEXT_SYNTHESIS` / `RISK_OPPORTUNITY_SYNTHESIS` | market/risk synthesis |
| `OUTREACH_DRAFT_GENERATION` | outreach-draft rendering |
| `TRANSLATION` / `LONG_CONTEXT_REVIEW` / `FAST_CLASSIFICATION` / `SAFETY_REVIEW` | general-purpose task slots |

`routeTask(taskType, configs)` in `router.ts` is a **pure function**: given the
requested task type and the list of `LLMProviderConfig` rows, it filters to
configs whose `taskTypesJson` includes that task, prefers `enabled` configs,
and breaks ties by `modelName` ascending. It returns `null` when no config
supports the task — routing never throws, it degrades to "nothing routed."

`LLMProviderConfig` carries only routing metadata — provider name, model
name, supported task types, cost/latency tier, enabled flag, an optional
fallback pointer. **It has no API-key column.** Keys live only in
`process.env.ANTHROPIC_API_KEY`, read once at call time and never persisted
or logged.

## 3. The dormancy model

`getActiveProvider()` (`provider.ts`) is the single source of truth for "is
orchestration live." It returns a usable provider only when **both**:

1. `process.env.ANTHROPIC_API_KEY` is set, **and**
2. at least one `LLMProviderConfig` row has `enabled: true`.

With the seeded (all-disabled) configs and no key present — the shipped
default — this always resolves to `null`. No network call is ever attempted
in that state; every code path that depends on `getActiveProvider()` treats
`null` as "run the deterministic path only."

`GET /api/llm/status` (`src/app/api/llm/status/route.ts`) reports this
without ever leaking the key:

```json
{ "configured": false, "activeProvider": null, "enabledTaskTypes": [] }
```

The `/opportunities/[id]` page and `/admin/llm` page both read this endpoint
(or the equivalent server-side check) to decide whether to show the
"AI enrichment not configured — deterministic playbook" note.

## 4. The validation pipeline (fail closed)

`runLLMTask(req, opts)` (`src/server/llm/run.ts`) is the only place an LLM is
ever actually called. It:

1. Resolves the provider (`opts.provider` if given — including explicit
   `null` to force dormant — otherwise `getActiveProvider()`).
2. If no provider: logs an `LLMRun` row with `status: 'SKIPPED_NO_PROVIDER'`
   and returns immediately. **No network call.**
3. If a provider exists: calls `provider.generate(req)`. A thrown error logs
   `status: 'FAILED'` and returns no text.
4. On a successful call, runs `validateLLMOutput(text, opts.validate)`
   (`src/server/llm/validate.ts`), which checks, in order:
   - **Schema** — if a Zod schema was supplied, the raw text must parse as
     JSON and satisfy it. No schema means this gate is trivially satisfied.
   - **Advice-language guard** — `findAdviceLanguage()`
     (`src/server/safety/advice-language.ts`) scans for buy/sell/hold framing,
     guaranteed-return language, price targets, etc. Any match fails the
     output outright.
   - **Evidence grounding** — if `requireGrounding` is set, at least one of
     the supplied `evidenceIds` (claim ids) must appear in the raw text.
     Ungrounded output fails.
5. `validationStatus` is `PASSED` only if all three gates pass; otherwise
   `FAILED`, and the run is logged as `REJECTED_VALIDATION`.
6. **`text`/`parsed` are only ever returned when `validationStatus === 'PASSED'`.**
   Callers cannot accidentally use a rejected output — the fail-closed
   contract is enforced by the return type, not by caller discipline.
7. A rejected output's raw text is never retained: `LLMRun.outputSummary` is
   replaced with `[redacted: output failed validation]` when validation
   fails, so a blocked "buy this stock" attempt never lingers in the audit
   trail.

## 5. Playbook enrichment specifically

`generatePlaybook(cardId, opts)` (`src/server/playbook/service.ts`) is the
concrete instance of this pattern for `OpportunityPlaybook`:

1. **Always** builds the deterministic baseline first — `targetBuyer` from
   `likelyBuyers[0]`, `painStatement` from the card's `buyerPain`,
   `offerAngle` from `suggestedOffer`, a composed `commercialHypothesis`,
   `discoveryQuestions`/`likelyObjections`/`proofPoints` from
   per-`opportunityType` templates, `firstAction` from `nextBestAction`,
   `confidence` copied from the card. Every field is checked by both
   `assertNoAdviceLanguage` and an additional narrow
   "no guaranteed outcome" guard (`findGuaranteedOutcomeLanguage` — blocks
   "guaranteed conversion", "will close", "guaranteed"). This is persisted
   via an **upsert unique on `opportunityCardId`** — regenerating never
   duplicates.
2. **Then**, if a provider is active (injected or from `getActiveProvider()`),
   calls `runLLMTask('OPPORTUNITY_PLAYBOOK_GENERATION', ...)` with
   `validate: { schema: PlaybookSchema, evidenceIds: [...claim ids from the
   event's evidence], requireGrounding: true }`.
3. Only when that run's `status === 'SUCCEEDED'` **and** the parsed JSON
   independently passes the playbook's own guard-clean check on every text
   field does the playbook get overwritten and `generatedBy` flip to `'LLM'`.
   Any failure at any stage — schema mismatch, ungrounded text, advice
   language, a guaranteed-outcome phrase the general advice guard doesn't
   catch — leaves the deterministic playbook untouched.

No invented account/contact data ever enters a playbook: every field is
either copied from the `OpportunityCard`/`EventCandidate` it was generated
from, or (in the LLM path) required to reference the same evidence claim ids
that back that card.

## 6. The audit trail

Two tables record every attempt, whether dormant, successful, or rejected:

- **`LLMRun`** — one row per `runLLMTask` call: task type, provider/model,
  a `promptHash` (sha256 of system+prompt — the raw text is **never**
  persisted), a non-sensitive `inputSummary` (character-length shape only),
  `outputSummary` (redacted on rejection), status, token counts, an
  estimated cost, latency, and an error message on failure.
- **`LLMOutputValidation`** — one row per validated run: which gate(s)
  passed/failed (`schemaValid`, `evidenceGrounded`,
  `prohibitedLanguageDetected`, `unsupportedClaimsDetected`) and free-text
  `reviewNotes` summarising why.

`getLLMAudit(limit)` (`src/server/services/playbook.ts`) serves both tables
to the read-only `/admin/llm` page: provider configs (name, model, enabled,
supported task types — no key), and the most recent `LLMRun` rows joined to
their validation outcome.

## 7. How to activate

Orchestration ships dormant. To turn it on:

1. Set `ANTHROPIC_API_KEY` in the environment.
2. Install the optional peer dependency: `npm i @anthropic-ai/sdk`. The
   import in `provider.ts` is lazy and guarded — without the key set, the
   package is never imported at all; without the package installed, calls
   degrade to `NoProviderConfiguredError` instead of crashing.
3. Enable at least one `LLMProviderConfig` row for the task type(s) you want
   live (e.g. flip `enabled: true` on a config whose `taskTypesJson` includes
   `"OPPORTUNITY_PLAYBOOK_GENERATION"`).

With both the key set and a config enabled, `getActiveProvider()` returns the
Anthropic-backed provider and `/api/llm/status` reports `configured: true`.
Every existing safety gate (schema, advice-language, grounding, audit
logging) applies identically to live calls — nothing about activation
bypasses validation.

## 8. Safety guarantees

- **No key → no network call, ever.** Verified by tests injecting
  `provider: null` and by `getActiveProvider()`'s short-circuit on a missing
  key.
- **Fail closed.** `runLLMTask` only returns usable text on `PASSED`
  validation; playbook generation only upgrades `generatedBy` to `'LLM'`
  after a second, independent guard-clean check on the parsed output.
- **No secrets logged.** `LLMRun` stores a prompt hash and shape-only
  summaries, never raw prompt/system text or the API key.
- **No advice, no guarantees.** Every generated text field — deterministic or
  LLM-upgraded — passes the shared advice-language guard
  (`src/server/safety/advice-language.ts`) plus the playbook-specific
  guaranteed-outcome check.
- **Evidence-linked, not invented.** LLM-upgraded playbook fields must ground
  in the same claim ids drawn from the opportunity's underlying event
  evidence; nothing is required to be an entirely new fact, and no
  account/contact data is fabricated anywhere in the pipeline.

## 9. What's deferred (not gaps)

- **Live model calls** are the owner's decision — nothing here calls a real
  API in tests or by default; activation is a manual, explicit step (§7).
- **Market-context synthesis** (`MARKET_CONTEXT_SYNTHESIS`,
  `RISK_OPPORTUNITY_SYNTHESIS`) is routed but not yet wired to a generating
  service — tracked for Phase 3e.
- **Portfolio-level "save"/comparison workflows** across multiple playbooks
  are tracked for Phase 3f, not this phase.
